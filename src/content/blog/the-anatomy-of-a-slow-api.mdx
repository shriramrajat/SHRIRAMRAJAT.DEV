I was completely wrong.

Once I passed 50 concurrent connections, the latency trajectory didn't shift linearly. It spiked exponentially.

```json

```

### The Invisible Bottleneck

What happens when you execute `SELECT * FROM users WHERE city = 'London';`?

If `city` is not indexed, PostgreSQL has almost zero contextual awareness. It executes a **Sequential Scan** (Full Table Scan), iterating O(n) checking 100,000 distinct permutations. 

Under single connection conditions, this hides under the guise of an acceptable "15ms query". Under concurrent load, the read buffer I/O hits a ceiling and aggressively queues requests.

### The Indexing Shift

Applying a B-Tree index reorganizes the search path entirely.

```sql
CREATE INDEX idx_users_city ON users(city);
```

By switching execution from `Sequential Scan` to an `Index Scan`, we shifted the time complexity boundary from linear `O(n)` to logarithmic `O(log n)`. 

The response? 
**979ms to 2.91ms**. 

It’s completely ruthless. A 98.5% latency destruction just by understanding how the hardware is retrieving blocks under the hood. Stop blaming your API frameworks; index your read engines.
